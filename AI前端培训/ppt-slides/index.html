02. 能力边界与适用场景

✅ 放权给 AI (脏活累活)

  • 骨架搭建：根据 Figma 图层反向堆页面和写样式
  • 批量平替：组件库迁移、JS一键转TS、API接口替换
  • 日志查错：一长串控制台报错的调用栈分析
  • 文档辅助：写注释、边界用例枚举、正则

⚠️ 死抓人工主导 (核心卡口)

  • 需求把脉：项目该做成什么样的人能判断，AI听不懂业务隐喻
  • 底层架构：核心组件抽象方案与选型拍板
防沉迷准则： 当 AI 15 分钟无产出时 → 立刻切回人工接管。

03. 三大核心抓手

掌握它们，AI 就从“野生工具”变成“规范资产”。

1. RAG (提供项目事实背景)

利用 @文件 或接入接口平台文档，让 AI 在改代码前“读懂”当前目录业务契约与技术规范。

2. Skill (规范指令锁死)

把团队“不要滥用any、必须用Element样式”的经验固化成一段系统指令（Prompt），强制 AI 遵守。

3. Workflow (标准流程推进)

编排人机协作链路序列：“需求理解 -> 写测试纲领 -> 依据测试纲领写业务”。按流水线执行。

01. 旧习惯 vs 新型 6 步工作流

过去痛点

口述需求，边猜边写

  • 理解偏差导致的大规模二次返工
  • 代码样板手搓耗时长、重复率高
  • 联调看心情：接口文档不同步，全是 console.log 查错
新范式引擎

AI 增强型 6 步推进法

  1. 人工主导： 需求与方案架构设计
  2. 团队对齐： 确认技术实施方案再动工
  3. 颗粒拆解： 精细化 Task 任务拆解 (极易跳过)
  4. AI 编码： 一 Task 一 commit，即插即验证
  5. 即时联调： MCP 直接拉取最新 Swagger 结构对接
  6. 人工审查： 严肃的 Code Review 及质量把控

02. 实战案例 A：从零建页演示

从一份 Figma 和一份 ApiFox Schema 开始：

Step 1 需求喂入
告诉 AI 业务实体是什么，引入页面结构大纲。
Step 2 抽取骨架
让 AI 基于 UI 稿结构，搭出前端基础组件树。
Step 3 获取契约并直连
RAG 输入 API 描述，一次性写清接口返回体拦截与异常气泡。

爽点： 前端无需等服务器发版，靠 Schema 就完成了高拟真联调闭环。

03. 实战案例 B：存量组件安全重构

如何干掉一个历史欠账严重的 500 行组件？

  1. 全盘拉取： 先让 AI 扫读整个上下文，生成现有组件树依赖图。
  2. 重构建议： 让 AI 先出“切分计划” 【人工评审该计划】
  3. 断点续传： AI 逐块（State、View、API）单独生成切块代码。
  4. 边界护航： AI 自动补齐极端的边界情况用例验证。

01. 致命操作避坑 (Top 3 翻车重灾区)

🔥 需求不清就撒手

需求文档没想透，就让 AI 写。垃圾进必然垃圾出。AI 给错了方向导致推翻重来的隐形成本极其高昂。

🍔 一块塞太多任务

一次对话指令给 AI 派发 10 个修改点任务，极易造成上下文被截断和幻觉代码插入，颗粒度一定要细。

⚠️ 无卡口的盲目自信

对 AI 输出不做验证直接 Git Commit。必须有严密的人工 Code Review：边界用例是否安全、可维护性是否恶化。

02. 架构与控制权陷阱

你的项目控制权正在被悄然“剥夺”吗？

  • AI 的万金油综合症： 它倾向拿市面上最时髦或通用套路架构直接套你代码，且无视当前项目的历史负债。
  • 黑盒代码积累： 一直用 AI 开发却无人懂其原理的组件连跑 3 个月后，突然宕机却无人能看懂它写的魔法数字。

架构由人主导： 把可维护性放在与正确性同等位置审查，哪怕是用 AI 生成的代码，你也必须“说得出每一行的理由”！

03. 从个人摸索走向团队资产

摒弃“闭门造车”

一个好用的 Prompt 是被无数坑喂出来的。如果不沉淀到项目的 .rules 库中，其他同事还将再遭受一遍痛苦。

[插入：.rules 规范约束或团队沉淀系统库截图]

推进落地方法论：

  1. 个人破冰： 自己先跑通闭环工作链路。
  2. 提炼精华： 把成功解法抽为 2-3 个标准化 Workflow 和 Skill。
  3. 星火分享： 只发给感兴趣的人先行测试，不要一锅端强制落地。
  4. 小步快跑： 根据业务的反馈，半年时间团队内自发跃迁。
