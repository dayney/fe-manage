- 一、AI 辅助前端概述与学习路径
  - 对齐认知：AI 发展与工具谱系
    - 发展脉络：补全式（Copilot）→ 会话式（通义灵码/等）→ 工程化（Antigravity + RAG/MCP）
    - 关键变化：更长上下文、多模态、全局编辑/批量改造、可编排工具调用
    - 使用体验对比：
      - Copilot：内联补全强、门槛低；项目级上下文/全局改造较弱
      - 通义灵码（示例）：中文场景友好、云生态联动；跨项目编排待验证
      - Antigravity：Chat → Composer → Edits → Diff/Commit 的端到端闭环，项目级上下文，批量重构，支持 MCP 扩展
    - 为什么落点在 Antigravity：覆盖"从需求确认到合并"的完整链路，对新页搭建/重构/联调/测试显著提效
  - 适用场景与边界（何时用AI、何时人工）
    - 基本定位
      - AI 用来"帮忙写代码"和做确定性任务
      - 项目架构与程序设计由人工主导与拍板
      - **没有思路时把 AI 当"前辈"咨询 → 先理解方案再实施**
      - 使用 AI 的人对产出代码负第一责任
    - 适用（AI优先）
      - 页面/组件骨架、样式初稿、路由与脚手架
      - 重构与迁移：命名优化、拆组件、API 替换、TS 化
        - 📖 参考：[前端工程化实践 Blog](https://dayney.github.io/frontend/engineering/)（我自己整理的工程化笔记，可作为补充阅读）
      - 联调与排错：接口对接、控制台报错分析、网络请求异常定位、正则表达式
      - 文档与说明：README、变更说明、代码注释、边界用例枚举
      - 设计到代码：Figma 图层解析 → 组件/样式映射
    - 不适用（人工优先）
      - 核心业务规则与关键路径决策
      - **架构设计、技术选型（必须人工拍板）**
      - 需求理解与方案确认（AI 理解需求存在偏差风险）
      - 难题攻关与权衡（一致性/可维护性/扩展性）
    - 决策准则
      - 低风险、重复性强、输入输出清晰 → AI 先行
      - 高风险、强约束、长期维护 → 人工主导，AI 辅助
      - 时间盒：AI 10–15 分钟无有效产物 → 人工接管/换思路，可定时让AI汇报进度
    - 质量流程（必须）
      - AI 产出 → 本地验证 → 人工 Code Review → 合并
      - Review 重点：正确性（边界/异常/并发）、可维护性（命名/抽象/注释）、性能基本面、业务逻辑是否正确
      - PR 需注明：AI 参与范围与验证方式（便于后续迭代与修复）
  - 学习路径与评估：范式 → 工具 → 实战 → 评测
    - 1. 范式 (Paradigm)：转变思维模式
      - 核心思想：从"动手编码者"转变为"指挥者和质检员"。主要工作不再是逐行写代码，而是将需求清晰地拆解，并用结构化的语言（Prompt）指挥 AI 完成
      - 关键转变：
        - 描述驱动开发：用清晰的指令代替手动编码，让 AI 生成骨架和模板
        - 迭代式优化：你的角色是审查、测试和优化 AI 的产出，而不是从零开始
        - 人机协作：你负责架构设计、核心逻辑和最终决策；AI 负责执行重复性、确定性强的编码任务
    - 2. 工具 (Tools)：掌握核心武器
      - 核心思想：选择并熟练掌握能够将新范式落地的工具，形成高效的工作流
      - 关键工具与流程：
        - IDE 助手（Antigravity）：打通从"对话 (Chat)"生成代码，到"编辑 (Edits)"和"提交 (Commit)"的完整开发闭环
        - Prompt 编写：学习如何写出高质量的指令（角色、约束、示例），这是指挥 AI 的核心技能
        - 上下文管理 (RAG)：让 AI "读懂"你的项目文档、接口定义，使其产出更贴合项目实际
    - 3. 实战 (Practice)：在真实场景中应用
      - 核心思想：将新的思维范式和工具应用到日常开发的具体任务中，形成肌肉记忆
      - 典型应用场景：
        - 页面/组件搭建：快速生成符合需求的 UI 骨架和样式
        - 代码重构与迁移：批量修改命名、替换 API、升级技术栈（如 JS 转 TS）
        - 联调与排错：快速生成接口对接代码、分析错误日志、编写正则表达式
        - 文档与测试：生成代码注释、README、变更说明，以及单元测试用例
    - 4. 评测 (Evaluation)：衡量效果与持续改进
      - 核心思想：建立一套标准来衡量 AI 辅助工作的成效，并确保代码质量，同时反思和提升自己的能力
      - 评估维度：
        - 效率评估：AI 是否显著节省了开发时间？在哪些环节提效最明显？
        - 质量评估：AI 生成的代码是否通过了 Code Review、Lint 检查和自动化测试？是否引入了潜在的维护性问题？
        - 能力评估：你的 Prompt 编写能力和代码审查能力是否在提升？能否快速识别并修正 AI 的错误？
  - 展望与趋势（多模态、AI Agent）
    - 多模态能力的工程化落地
      - 设计稿直接转代码（Figma → 组件 + 样式，通过 Figma MCP 实现）
      - 截图/草图理解（手绘原型 → 可运行页面）
    - AI Agent 在前端工程中的应用
      - 自主任务执行：给定需求 → AI 自动拆解 → 编写代码 → 运行测试 → 提交 PR
      - 跨工具协同：IDE + 浏览器 DevTools + API 文档 + 监控平台的打通
      - 持续学习：从团队 PR、Code Review 记录中学习代码规范和业务模式
    - 对前端架构师的影响
      - 核心价值转移：从"写代码"到"定义规范、设计架构、把控质量"
      - 新技能要求：Prompt 工程、AI 产出评审、人机协作流程设计
      - 团队管理变化：如何培养团队的 AI 协作能力、如何评估 AI 辅助的 ROI

- 二、核心方法与工具（Antigravity / Prompt / RAG）
  - Antigravity 工作流：Chat → Composer → Edits → Diff/Commit
    - Chat（聊天面板）：用于连续对话、讨论思路、确认方案、获取上下文
    - Composer（Cmd/Ctrl + K）：用于执行目标明确、一次性的代码生成或编辑任务
    - Edits（编辑预览）：AI 执行指令后，以可预览的 diff 形式呈现修改，供你确认
    - Diff/Commit（提交）：接受修改后，通过 Git 工具进行比对和提交
  - Cursor vs Antigravity：从 Cursor 迁移的认知对齐
    - **为什么要做对比**
      - 基于我自己的使用经验：两者核心理念相通，从 Cursor 迁移到 Antigravity 成本很低
      - 概念名称和配置方式有差异，对齐后可以快速上手
    - **工作流对比**
      | 能力 | Cursor | Antigravity |
      |---|---|---|
      | 对话生成代码 | Chat 面板 | Chat 面板 |
      | 快速内联编辑 | Cmd+K（Composer）| Cmd+K（Composer）|
      | 多文件批量编辑 | Cursor Composer | Edits 模式 |
      | 变更预览 | Diff 视图 | Diff 视图 |
      | Git 提交 | 终端 / 源码管理 | Diff/Commit 集成 |
      | MCP 扩展 | ✅ 支持 | ✅ 支持（更完整）|
    - **规则配置对比（Rules）**
      - **核心价值**：Rule 的本质是**缩小 AI 的检索范围，提高 AI 与当前项目的匹配精度**——让 AI 不再对每个项目"零基础"开始，而是一开口就已经了解你的技术栈、规范约定和禁止事项
      - **Cursor** 的 Rule 机制
        - **`.cursorrules`**：放在项目根目录，内容始终注入到每次对话，全局生效
      - **Antigravity** 的 Rule 机制
        - **`.rules`** 文件（放在项目根目录）：与 `.cursorrules` 功能完全对应，**内容直接迁移即可**
        - 核心作用：全局规范注入、每次对话自动携带、无需重复说明约束
        - 示例内容：技术栈（Vue3+TS+Element Plus）、命名规范、禁止使用 `any`、AI协作约定
      - **一句话对应关系**：`.cursorrules` = `.rules`，直接改名迁移即可
  - Prompt 编写技巧：角色/约束/少样本/结构化输出
    - 核心技巧速查：
      - **角色 (Role)**：明确 AI 的身份，如「你是一位资深前端工程师，精通 React 和 TypeScript」
        - 在 Cursor/Antigravity 中可通过 `.cursor/settings.json` 的 `Template.chatSystemPrompt` 全局设定，无需每次重复
      - **约束 (Constraints)**：限定 AI 只在本项目的技术栈和场景内思考，而不是漫无边际地给建议
        - 实践做法：在项目根目录 `docs/` 目录中放入技术栈说明，通过 @引用 让 AI 了解本项目的技术边界
        - 例：「只使用 Vue 3 + Element Plus，不引入任何新依赖，遵循项目已有的命名规范」
      - **少样本 (Few-shot)**：提供 1-3 个示例，让 AI 理解期望的输入输出格式
      - **结构化输出 (Structured Output)**：要求 JSON/Markdown/XML 格式，便于后续解析和处理
      - **思维链 (Chain-of-Thought)**：要求 AI "一步步思考"，提高复杂任务的准确性
      - **上下文引用**：在 Antigravity 中使用 @文件名、@文件夹、@代码块来精准提供上下文
  - RAG 与代码上下文：项目文档/接口/设计接入与检索
    - 定义：RAG（检索增强生成）= 让 AI 先读取项目真实资料（接口文档/技术规范/设计稿），再在**本项目的技术范围内**生成代码，而不是根据模型记忆漫无边际地瞎想
    - 为什么需要：解决知识过时、不了解本项目、缺少业务上下文三大痛点
    - 可接入的上下文源：
      - 项目文档：README/编码规范/组件指南 → 产出符合团队约定的实现与用法
      - 接口文档：ApiFox（通过 ApiFox MCP 直接读取接口定义）→ 产出正确的请求方法、参数/类型、错误处理
      - 设计资料：Design Tokens/Figma → 产出符合设计系统的结构与样式
    - 使用方式（Antigravity）：
      - 本地 RAG：用 @文件/@文件夹/@符号 精准引用上下文（例：@api/openapi.json、@docs/guide.md）
      - 云端 RAG（MCP）：接入 ApiFox/Figma/Chrome DevTools 等 MCP 服务器，实时检索外部资料并返回给模型
    - 典型流程：提出任务 → 检索最相关上下文 → 将上下文与指令一并送入模型 → 生成代码 → 本地验证与 Review
    - 与 Rule 的区别：Rule 是长期"怎么做"的硬性规范；RAG 是当前任务"做什么"的事实依据，二者互补（冲突时以 Rule 为准）
    - 一句话总结：用规则统一风格，用 RAG 对齐事实，让 AI 既"会做"，又"做对"
  - 质量与评测：Prompt 回归、lint/typecheck/测试流水线结合
    - 定义：为 AI 代码建立"双重质检"，既要审查"最终产物"，也要测试"生产指令"
    - 产出质检（防守）：将 lint、类型检查、自动化测试等现有流水线，应用于 AI 生成的代码，保证质量下限
    - 指令质检（进攻）：通过"Prompt 回归"测试，确保核心 Prompt 在迭代后，产出质量不下降，提升质量上限
    - 价值：从"被动修复"到"主动预防"，系统性地提升 AI 辅助开发的可靠性
  - Skill 使用技巧：将专家经验沉淀为可复用的 AI 指令模块
    - **什么是 Skill**
      - Skill 是一个放在项目 `.agents/skills/` 目录下的 Markdown 文件，包含 YAML 前置信息 + 详细指令
      - 作用：把团队的最佳实践（规范、风格、流程）封装成"一句话激活"的 AI 指令模块
      - 与 Rule 的区别：Rule 是项目级的全局约束（始终生效）；Skill 是按需调用（需要时激活）
    - **实战案例：`frontend-design` Skill**
      - 文件：`.agents/skills/frontend-design-skill.md`
      - 解决的问题：AI 生成 UI 默认风格平庸单一（"AI味"浓），缺乏真正的审美个性
      - 核心能力：激活后强制 AI 遵循高定级别的前端美学准则（排版、色彩、动效、空间构图）
      - 使用方式：在 Prompt 中直接声明风格意图即可激活
        - 极简奢侈风：「请按照 `frontend-design` skill 的标准，采用 **luxury/refined** 风格重构组件，克制留白 + 高级字体 + 细致边框 + 优雅的 CSS 微交互，不使用渐变色」
        - 先锋杂志风：「使用 `frontend-design` 开启 **Editorial/Magazine** 风格，不对称布局 + 重叠元素 + 巨型衬线字体作为背景」
        - 工业机能风：「调用 `frontend-design`，采用 **Industrial/Utilitarian** 风格，几何分割 + 噪点质感 + 等宽字体」
    - **如何创建自己团队的 Skill**
      - 第一步：找到团队中"每次都要重复说"的规范 → 这就是 Skill 的雏形
        - 来源1：**重复的口头约束** — 例：每次开发表单组件，都要提醒 AI "用 Element Plus + 统一 labelWidth + 必须处理 loading 状态"
        - 来源2：**项目现有的 SOP 操作文档** — 梳理其中重复的公共步骤，抽离成通用描述，即可作为 Skill 的初稿
          - 例：部署流程 SOP → 抽离出"每次发版前必须做的检查项" → `pre-release-check` Skill
          - 例：代码审查 SOP → 抽离出"Review 必检维度" → `code-review` Skill
        - 来源3：**踩过的坑** — 每次返工的根因，往往就是缺少一条约束，把它加进 Skill 里
      - 第二步：将这些约束写成清晰的 Skill 文件（参考 `frontend-design-skill.md` 的结构）
        - Skill 正文：核心理念 → 使用指令模板 → 遵循的细则 → 验收标准
      - 第三步：在 Prompt 中 @引用 Skill 文件，或直接在描述中提及 skill 名称激活
      - 第四步：沉淀到团队仓库，持续迭代优化
    - **Skill 的核心价值**
      - **一次编写，团队复用**：高级工程师的经验不再只存在于个人脑子里
      - **降低沟通成本**：新人用 Skill 也能产出符合团队规范的代码，减少 Review 反复修改
      - **保持风格一致性**：多人协作时 AI 产出风格统一，不因人而异
      - **可迭代积累**：随项目成熟，Skill 越来越精准，AI 产出质量持续提升
    - **推荐的团队 Skill 清单（可逐步建立）**
      - `frontend-design` — UI 高阶审美规范（已有案例）
      - `api-integration` — 接口对接规范（ApiFox MCP + useRequest + 统一错误处理）
      - `component-standard` — 组件开发规范（props 定义、emit 规范、注释要求）
      - `code-review` — AI 辅助 Code Review Checklist
      - `git-commit` — Git Commit 规范（类型 + 范围 + 描述格式）
  - Workflow 使用技巧：把研发流程封装成 AI 可执行的标准流程
    - **什么是 Workflow**
      - Workflow 是放在 `.agents/workflows/` 目录下的 Markdown 文件
      - 作用：把多步骤的操作流程封装成 AI 可逐步执行的"剧本"，一句话触发
      - 与 Skill 的区别：Skill 是"规范约束"；Workflow 是"步骤流程"——前者约束产出质量，后者驱动执行过程
    - **AI 对工作流的理解**
      - AI 读取 Workflow 文件后，会**逐步执行**每个步骤，并在关键节点等待人工确认
      - AI 能理解步骤间的依赖关系：上一步的产物作为下一步的输入
      - 标注 `// turbo` 的步骤可自动执行（无需人工确认），提高自动化程度
      - AI 执行中遇到歧义会主动提问，而不是自行猜测（避免方向跑偏）
    - **是否要把研发流程封装成 Workflow？— 强烈建议！**
      - 第三章我们梳理了新的研发流程（6个阶段），这正是 Workflow 的最佳适用场景
      - 好处：
        - 新人加入团队，直接用 Workflow 就能按规范走完整个流程
        - 每次启动新需求，不需要靠记忆回忆"下一步该做什么"
        - 把团队的最佳实践固化下来，不因人员变动而流失
    - **推荐封装的典型 Workflow**
      - `new-feature.md` — 新功能开发流程（需求澄清 → 方案设计 → 编码 → 联调 → Review）
        - 步骤1：让 AI 复述需求理解，列出问题清单（人工确认后继续）
        - 步骤2：生成技术方案文档，等待团队评审确认
        - 步骤3：按子任务拆分逐步编码，每步本地验证
        - 步骤4：联调阶段引用 ApiFox MCP
        - 步骤5：生成 PR 描述，跑 Lint/TypeCheck
      - `code-review.md` — AI 辅助 Code Review 流程
        - 步骤1：@引用待 Review 的文件，AI 按 Checklist 逐项检查
        - 步骤2：AI 列出问题清单（分类：必须修改 / 建议优化 / 可忽略）
        - 步骤3：生成 Review 意见，附上修改建议代码
      - `git-commit.md` — 规范化 Git 提交（已有案例）
        - 自动分析变更内容，生成符合 Conventional Commits 规范的 commit message
      - `bug-fix.md` — Bug 修复流程（问题复现 → 定位 → 修复 → 回归测试）
    - **Workflow 文件结构（简要示意）**
      ```markdown
      ---
      description: 新功能开发标准流程
      ---
      1. 需求澄清：让 AI 复述需求，列出疑问清单，等待人工确认
      // turbo
      2. 生成技术方案文档（组件树 + 接口依赖 + 状态设计）
      3. 方案评审确认（人工介入，拍板后继续）
      // turbo
      4. 按子任务逐步编码，每步验证
      5. 生成 PR 描述并运行质量检查
      ```
    - **Workflow 与 Skill 的协同使用**
      - Workflow 驱动流程（做什么、按什么顺序）
      - Skill 保障质量（怎么做、产出什么标准）
      - 例：`new-feature` Workflow 执行到"编码"步骤时，自动调用 `component-standard` Skill 约束产出规范
  - 【影响对比】无AI  vs  有AI  vs  有AI + 多Workflow
    - **阶段一：常规开发（无 AI）**
      - 每个环节完全依赖人工：需求理解靠沟通，编码靠经验，联调靠反复试错
      - 流程规范停留在文档里，执行靠个人自觉，难以保证一致性
      - 新人上手慢，经验传递靠"传帮带"，效率瓶颈明显
      - **典型痛点**：骨架代码手写耗时、重复性工作多、规范难以落地、交付周期长
    - **阶段二：引入 AI（有AI，无系统化配置）**
      - AI 可以帮助加速编码，减少重复性工作
      - 但每次使用需重新描述约束，产出风格不稳定（因人而异）
      - 团队各成员 Prompt 水平参差不齐，AI 使用效果差异大
      - **收益**：编码速度提升 40-60%，但需求理解/联调/规范一致性问题依然存在
      - **局限**：AI 是"个人工具"，无法系统性提升团队整体效率
    - **阶段三：AI + 多 Workflow（体系化落地）**
      - **Rules**：保证每次对话都遵守团队规范，无需每次重复说明
      - **Skill**：专项场景一句话激活专家级约束（UI设计/接口规范/代码规范）
      - **Workflow**：把整个研发流程封装成 AI 可执行的剧本，新人也能按规范走完全流程
      - **MCP**：打通外部工具（Figma/ApiFox/DevTools），AI 直接读取真实数据源
      - **收益叠加**：
        - 流程规范自动执行，不再依赖个人自觉
        - 新人上手时间从数周缩短到数天
        - 团队产出风格高度一致，Review 效率大幅提升
        - 最佳实践固化积累，随团队成长持续增强
    - **三态量化对比（参考数据）**

      | 维度 | 无 AI | 有 AI（无配置）| 有 AI + 多 Workflow |
      |---|---|---|---|
      | 新功能开发周期 | 基准 100% | ~60%（节省 40%）| ~35%（节省 65%）|
      | 新人达到效能时间 | 4-8 周 | 3-5 周 | 1-2 周 |
      | 返工率 | 高（需求/规范偏差多）| 中（方向偶有跑偏）| 低（流程固化，方向一致）|
      | 工作风格一致性 | 因人而异（低）| 依赖个人 Prompt（中）| Workflow 保障（高）|
      | 团队知识沉淀 | 靠文档（易过时）| 靠个人（不可共享）| Workflow/Skill 持续积累（越用越强）|
      | AI 账号成本 | 无 | 有 | 有（ROI 随积累持续提升）|

    - **核心观点：降本增效的本质是持续完善 Workflow/Skill 体系**
      - **不是用 AI 写代码更快** — 这只是表层收益，且会因人而异
      - **而是把最佳实践固化** — 让每一次踩坑都转化为下一次的规范保障
      - 每完善一个 Workflow，就减少一类返工；每沉淀一个 Skill，就统一一类产出风格
      - **复利效应**：Workflow/Skill 越积越多 → 覆盖场景越来越全 → 整体效率持续提升
      - 最终目标：不是"AI 帮我写了多少行代码"，而是"团队有没有一套可靠的 AI 协作体系"
    - **Workflow/Skill 持续积累的正向循环**
      - 踩坑 → 识别根因 → 封装成 Workflow/Skill → 下次同类场景自动规避
      - 新人加入 → 用现有 Workflow 上手 → 产出即符合规范 → Review 负担降低
      - 业务增长 → 场景增多 → Workflow 库扩充 → 总体交付能力持续增强
      - ⚠️ **关键**：Workflow 不是一次性配置，而是需要团队持续优化迭代的"生产资产"

- 三、我的 AI 工作流实践：从旧习惯到新方式

  > **我自己踩过的最大的坑：需求没搞清楚就让 AI 写，结果返工量等同于重写。现在我的铁律是：先沟通清楚需求，确认方案，再开发。AI 是执行高手，方向必须由我来定。**

  > [!IMPORTANT]
  > **AI 不只是让你"写代码更快"，它实际上重塑了整个前端开发流程：**
  > - **开发顺序变了**：从"边想边写"变成"先设计确认方案，再拆 Task，再执行"
  > - **联调方式变了**：不再靠反复问后端"这个字段是什么意思"——通过 ApiFox MCP，AI 直接读接口文档生成对接代码，前后端可以**真正并行**
  > - **交付节奏变了**：每个 Task 独立完成、独立提交，进度清晰可控，减少"一锅端"式的大合并
  > - **质量保障变了**：Skill/Rules 保证每次产出都有约束，不再靠人记得"应该怎么写"

  - 【旧流程】传统前端研发流程
    - 阶段1 需求接收
      - PM 口头/文档描述需求 → 开发理解可能存在偏差
      - 缺少可视化确认环节，需求歧义靠"猜"
    - 阶段2 方案设计（常被压缩/跳过）
      - 开发自行脑补技术方案
      - 缺乏团队对齐，各自为战
    - 阶段3 编码开发
      - 从零手写组件、样式、接口调用
      - 重复性工作占比高（样板代码、CRUD页面）
      - 依赖个人经验，质量参差不齐
    - 阶段4 联调测试
      - 接口文档版本不同步，频繁沟通后端
      - 问题定位靠 console.log，效率低
    - 阶段5 Code Review & 发布
      - Review 质量依赖reviewer经验
      - 发版流程手动，易遗漏
    - 典型痛点：**需求返工多 / 编码耗时长 / 联调摩擦大 / 质量不稳定**

  - 【新流程】AI 增强型前端研发流程
    - 阶段1 需求理解与方案设计（**人工主导，AI 辅助分析**）
      - 人工主导：与 PM 深度对齐，**明确边界、数据结构、交互逻辑、异常场景**
      - AI 辅助：将需求文档喂给 AI，让其帮助梳理功能点、识别潜在歧义、生成问题清单
      - 输出物：**方案设计文档**（页面结构、组件划分、接口依赖、状态设计）— 必须人工确认
      - ⚠️ **黄金原则：方案未确认，一行代码都不写**
    - 阶段2 技术方案确认（**团队对齐**）
      - AI 辅助生成技术方案初稿（组件树、数据流、接口契约）
      - **团队方案评审**（Walk-through，即所有人围在一起把技术方案文档逐项走读、讨论、确认，确保理解一致，相当于"技术方案评审会"），人工拍板最终方案
      - 确认后方案作为 RAG 上下文，后续开发全程参考
    - 阶段2.5 **任务拆解与计划制定**（⚠️ 常被跳过但非常重要）
      - 方案确认后，先让 AI 帮助将整个需求拆解为独立的小 Task（to-do list）
      - 每个 Task 需满足：**目标单一、输入输出清晰、可独立验证**
      - 产出：一份 Plan 文档或 task.md，明确排序和依赖关系
      - **为什么要拆 Task**：
        - 避免一次性给 AI 太大任务，导致上下文混乱、产出质量下降
        - 每个 Task 完成后立即 git commit，出问题随时可回滚
        - Plan 作为进度看板，清晰掌握开发进度
      - 示例拆解：「用户列表页」→
        - Task1: 页面路由和基础布局骨架
        - Task2: 搜索栏组件（UI + 状态）
        - Task3: 列表组件（UI + 分页）
        - Task4: ApiFox MCP 接入 + 接口联调
        - Task5: 边界处理（空数据/加载/错误状态）
    - 阶段3 AI 驱动编码（**AI 执行，人工审查**）
      - 用 Antigravity 驱动：将确认的方案和当前 Task 作为 Prompt 上下文
      - **每次只做一个 Task**，完成后本地跑通验证，再进行下一个
      - ✅ **每完成一个 Task → git commit 一次**（粒度要小，便于回溯）
      - 时间盒控制：单个任务 AI 15 分钟无有效产出 → 停下来拆得更细，人工介入
    - 阶段4 联调与问题定位（**MCP + AI 辅助**）
      - 用 ApiFox MCP 让 AI 直接读取接口文档，生成对接代码
      - 用 DevTools MCP 辅助网络请求分析、性能定位
      - AI 分析错误日志，给出修复建议（人工决策是否采纳）
    - 阶段5 Code Review & 质量卡口（**流程固化**）
      - Lint / TypeCheck / 单测 自动化检查（CI 强制卡口）
      - 人工 Review 重点：业务逻辑正确性、边界处理、可维护性
      - PR 模板必填：AI 参与的范围、验证方式、影响面评估
    - 阶段6 发布与复盘
      - AI 辅助生成 Changelog 和发布说明
      - 定期复盘：哪些场景AI提效明显？哪些翻车了？更新团队最佳实践

  - 【效率对比】关键指标变化（参考数据）
    | 环节 | 旧流程耗时占比 | 新流程耗时占比 | 预期提升 |
    |------|-------------|-------------|---------|
    | 方案设计 | 10%（常被压缩）| 20%（强制投入）| 减少后期返工 60%+ |
    | 骨架/样板代码 | 30% | 5% | 提效 80%+ |
    | 业务逻辑编码 | 35% | 40% | 质量提升，耗时持平 |
    | 联调排错 | 20% | 10% | 提效 50%+ |
    | Review & 修复 | 5% | 25%（故意加强）| 缺陷率降低 50%+ |

  - 【心态转变】为什么说 AI 是"助手"而非"替代者"
    - AI 的强项：执行力超强、永不疲倦、知识广博
    - AI 的弱项：不理解业务上下文、不懂团队潜规则、容易"自信地出错"
    - 正确心态：
      - ✅ 把 AI 当新来的实习生——你要给清晰的方向，并检查产出
      - ✅ 你的价值在于判断力：判断需求正确性、判断方案合理性、判断代码质量
      - ✅ AI 的成本是**固定的账号费用**，节省的是**时间 × 人工**——缩短整个交付周期，而不仅仅是编码环节
      - ✅ 交付提速 = 需求确认更快 + 编码更快 + 联调更快 + 文档自动生成，全链路加速
      - ❌ 不要把 AI 当"万能神"——需求模糊时 AI 生成的代码方向可能完全错误
      - ❌ 不要"AI 说啥就是啥"——AI 的自信程度与正确程度不成正比
    - **增强接纳率的关键**：不是"用AI写代码"，而是"用AI压缩交付周期"，你始终是主导者

- 四、我的实战经验分享与团队落地建议

  > **以下是我在日常工作中真实摸索出来的方法，不是教科书里的最佳实践，是我自己用着觉得有用的东西，分享给大家参考。**

  - 实操1：从需求确认到页面骨架与联调（Antigravity 驱动）
    - 演示场景：一个"用户列表页"，含搜索、分页、操作按钮
    - 步骤分解：
      1. **需求确认**：将 PRD 描述提供给 AI，让其复述理解、列出问题清单 → 发现3个歧义点 → 人工确认
      2. **方案设计**：AI 生成组件树（UserListPage / SearchBar / UserTable / Pagination）、接口依赖清单
      3. **骨架生成**：Antigravity Chat 中描述技术栈约束，生成组件骨架代码
      4. **接口联调**：先通过 ApiFox MCP 获取最新的接口 Schema，再让 AI 生成完整的请求/响应处理代码
      5. **逐步验证**：每步运行，发现问题立刻修正，不积压
    - 关键观察点：AI 如何处理不清晰的接口字段（AI 猜测 vs 正确引导）
    - 预期产出：可运行的列表页，含真实接口联调，约 30 分钟完成
    - 学员练习：给出一段模糊需求，让学员先澄清需求再给 AI，对比两种方式的产出差异

  - 实操2：重构与可维护性（组件抽象、状态、边界用例）
    - 演示场景：一个"功能堆砌"的老组件（500行，混合业务逻辑和 UI）
    - 步骤分解：
      1. **问题识别**：让 Antigravity 分析代码，列出可重构点（职责混乱/重复代码/命名不清）
      2. **拆分方案确认**：AI 生成重构方案 → 人工评审拍板 → 确认后再执行
      3. **渐进式重构**：按模块逐步重构，每步验证不破坏现有功能
      4. **状态管理优化**：识别可提升的状态，AI 辅助迁移至 Pinia/Zustand 等
      5. **边界用例补充**：让 AI 枚举边界场景（空数据/网络错误/并发请求），逐一处理
    - 关键观察点：重构过程中 AI 如何保持上下文一致性（避免越改越乱）
    - 学员练习：给出一段"有坏味道"的代码，用 AI 识别问题并给出重构建议

  - 实操3：端到端迭代（设计 → 开发 → 测试 → 评审）
    - 演示场景：**使用公司现有 Figma 设计稿**（选一个层级清晰的简单页面，如列表页/卡片组件）
    - **提前准备**：
      - 确认 Figma MCP 连接正常，提前测试读取是否顺畅
      - 选择命名规范、图层结构清晰的 Frame（乱命名的图层会影响演示效果）
      - 现场不追求生成完美代码，重点展示"从设计到骨架代码"的过程即可
    - 步骤分解：
      1. **设计解析**：用 Figma MCP 让 AI 读取设计稿，生成组件结构和 Design Token 映射
      2. **代码生成**：基于设计稿和确认的方案，Antigravity 生成完整页面代码
      3. **测试用例生成**：让 AI 根据组件功能，生成单元测试初稿（Vitest/Jest）
      4. **Lint & TypeCheck**：运行质量检查，AI 辅助修复问题
      5. **PR 撰写**：AI 生成 PR 描述（改动范围、影响面、测试说明、AI参与说明）
      6. **模拟 Code Review**：展示 Review Checklist 和常见问题
    - 关键观察点：从设计到 PR 的完整流程，展示 AI 在每个环节的具体价值
    - 学员练习：给出自己项目里的一个 Figma 截图，用 AI 生成对应的 Vue/React 组件骨架

  - 【真实案例复盘】开场展示的那个案例，是怎么做到的？
    - > 开场我展示了一组截图：Figma 设计稿 → ApiFox 接口文档 → 最终成品页面。这里还原当时完整的过程。
    - **输入**：Figma 设计稿（某个列表/详情页）+ ApiFox 上的接口文档
    - **完整流程**：
      1. 通过 Figma MCP 让 AI 读取设计稿，解析组件结构和样式
      2. 通过 ApiFox MCP 获取最新接口 Schema，AI 了解字段类型和响应结构
      3. 让 AI 生成组件骨架代码（已包含设计稿还原和接口定义）
      4. 逐步填充业务逻辑、边界处理、加载/错误状态
      5. 验证通过后 git commit，整个过程约 30-60 分钟
    - **与传统方式的差异**：
      - 传统：看设计稿 → 手写 HTML/CSS → 问后端字段含义 → 一点点联调
      - AI 方式：AI 同时读设计稿和接口文档 → 一次性生成有骨架的代码 → 直接在此基础上填充逻辑
    - **关键前提**：Figma 图层命名规范 + ApiFox 接口文档及时更新 → **AI 才能读懂**

  - 【我的最佳实践1】用 AI 快速熟悉陌生老项目（endiop-bi-platform-web）
    - **背景**：接手老项目是前端最头疼的事——没有文档、代码风格各异、业务逻辑全靠猜
    - **我现在的做法**：
      1. 让 AI 全文件扫读整个项目（@引用整个 src 目录）
      2. 让 AI 生成项目文档：模块划分、核心组件、数据流、接口依赖关系
      3. 基于文档让 AI 生成对应的 `.rules`（记录技术栈、命名约定、已有工具函数）
      4. 把文档和 `.rules` 作为后续所有开发的上下文基础，后续每次对话都携带
      5. 然后再分析需求、拆解任务、开始干活
    - **实际效果**：原来熟悉一个老项目需要 1-2 周，现在 1-2 天能有效上手
    - **额外收益**：AI 生成的文档比"问前人"更全面，还能顺手充当项目文档维护起来
    - **⚠️ 注意**：AI 的理解可能有偏差，关键业务逻辑必须人工核对

  - 【我的最佳实践2 · 扩展演示】用 AI + Playwright 写前端集成测试(admin-base v1.0.0)
    - > 🎯 **演示说明**：这是我用自己的项目摸索出来的方案，适合时间充裕时单独演示；时间紧可跳过，仅口头分享思路

- 五、坑与反模式（避坑指南）

  - 坑1：需求不清就让 AI 开发 ⭐⭐⭐（最高危）
    - 表现：把模糊的需求直接扔给 AI，期望它"猜对"
    - 后果：AI 生成的代码方向完全错误，改动量等同重写，大量返工
    - 正确做法：**先与 PM 对齐需求，写清楚输入输出和边界，再给 AI**
    - 口诀：**垃圾进，垃圾出（Garbage In, Garbage Out）**

  - 坑2：对 AI 产出不做验证，直接提交
    - 表现：AI 生成代码后，简单看一眼就 commit，没有运行测试
    - 后果：把 AI 的 bug 带入生产环境，Code Review 发现大量问题
    - 正确做法：每个子任务完成后立即本地验证，严格走 Review 流程

  - 坑3：一次性给 AI 做太多事
    - 表现：一个 Prompt 要求 AI 完成整个模块（十几个组件）
    - 后果：AI 产出质量快速下降，遗漏细节，需要反复纠正
    - 正确做法：**任务颗粒度要小**，每次让 AI 完成一个明确的子任务，验证通过再继续

  - 坑4：Context 污染（长对话导致 AI 偏移）
    - 表现：在同一个 Chat 中反复修改需求，AI 前后矛盾，越改越乱
    - 后果：AI 开始"忘记"之前的约束，产出代码风格不一致
    - 正确做法：**新任务开新 Chat**，将关键上下文（方案文档）重新提供，不要依赖长对话记忆

  - 坑5：AI 幻觉——错误但自信的输出
    - 表现：AI 编造了不存在的 API、库函数，信誓旦旦地给出错误代码
    - 典型场景：涉及版本较新的库、私有项目的特定业务逻辑
    - 识别方法：涉及具体 API/版本时，必须查阅官方文档验证
    - 正确做法：不熟悉的领域永远要交叉验证，不能完全信任 AI 的"自信"

  - 坑6：让 AI 做架构决策
    - 表现：让 AI 自行决定"用什么状态管理方案"、"怎么设计数据库结构"
    - 后果：AI 会给出"通用"答案，不符合团队实际约束，后续难以维护
    - 正确做法：架构决策由人主导，给 AI 明确约束后再让它执行

  - 坑7：团队 Prompt 没有沉淀
    - 表现：每个人自己摸索 Prompt，重复踩坑，经验不共享
    - 后果：AI 使用效果参差不齐，团队整体提效有限
    - 正确做法：建立团队 Prompt 模板库，将有效的 Prompt 模式文档化共享

  - 坑8：忽视 AI 参与的代码可维护性
    - 表现：AI 生成的代码通过了测试，但充满"魔法数字"、缺少注释、命名随意
    - 后果：3个月后没人看得懂，维护成本极高
    - 正确做法：Review 时把"可维护性"作为必检项，与正确性同等重要
